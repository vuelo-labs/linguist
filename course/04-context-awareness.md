# Layer 4 — Context Awareness
## Skills 10–12

Context awareness in 2026 is no longer just "watch the window fill up and brace for compaction". The window is bigger (1M tokens GA on Sonnet 4.6 and Opus 4.6 since 2026-03-13, at standard pricing), compaction has moved server-side as a first-class API primitive (Opus 4.6, beta as of 2026-05-05), and Claude Code skills can fork their own context via `context: fork`. The skill is choosing the right primitive — manual checkpoint, `/compact`, Compaction API, Memory, or forked-skill context — for the workload in front of you. These three skills cover the recognition, writing, and consolidation moves that make those choices well.

**The 2026 context-management primitives at a glance:**

| Primitive | Where it lives | What it costs | When to reach for it |
|-----------|----------------|---------------|----------------------|
| Manual checkpoint message | Your turn, in the conversation | One turn, fully under your control | Mid-task, before a hard sub-step, when you want a *verified* summary before continuing |
| `/compact` | Claude Code, client-side | Forces the existing autocompact path early | Window is filling but you're not done; you'd rather compact on your own timing than the harness's |
| Compaction API | Anthropic API, server-side (Opus 4.6, beta as of 2026-05-05) | API-side summarisation; "effectively infinite" conversations | Long-running API workloads where you want the platform to manage rolling summaries instead of hand-rolling them |
| Memory (Managed Agents) | File-backed store, scoped + audited (public beta as of 2026-05-05) | Cross-session state with permissions; not free — beta header + audit | State that should survive *between* sessions, not just within one |
| Forked-skill context (`context: fork` in `SKILL.md`) | Claude Code skill activation | A fresh sub-context for the skill; deferred tools (WebSearch / WebFetch) become available on first turn (v2.1.126) | A skill that needs clean working memory and/or web tools, without polluting the parent session |

A 200k-window mental model still applies to most Claude Code sessions (the harness has not defaulted to 1M). But for API workloads on Sonnet 4.6 / Opus 4.6 at 1M, and after the `context-1m-2025-08-07` beta header retired for Sonnet 4.5 / Sonnet 4 on 2026-04-30, the *urgency* of context-rot framing varies a lot by substrate. These skills are about reading your substrate, not about treating every session like a 200k window under siege.

**What compaction preserves** (from `services/compact/prompt.ts`, as of v2.1.128, 2026-05-05; structure verified, exact wording may have drifted):

1. Your explicit requests and intents
2. Key technical concepts and frameworks discussed
3. Specific file names, code snippets, and function signatures
4. Errors encountered and how they were fixed
5. Your feedback — especially corrections where you said "do it differently"
6. All of your messages, verbatim
7. Pending tasks
8. What was being worked on immediately before summarisation
9. The next step, with a direct quote to prevent drift

**What compaction loses:**
- Raw tool output (file reads, bash results, grep output)
- Intermediate model reasoning
- Implicit context established by example rather than stated explicitly
- Nuanced constraints expressed conversationally rather than as clear rules

The Compaction API (server-side, Opus 4.6, beta as of 2026-05-05) implements the same shape — it's the server-side counterpart to `/compact`, not a different summarisation strategy. The lossiness profile is similar: user messages survive well, tool output and implicit context don't. Where it differs is the lifecycle: the API rolls summaries continuously as part of the request, so you don't need a Claude Code harness in the loop. The skill of *writing for compaction* (Skill 11) transfers directly.

### Skill 10: Recognising Context Rot

**What it is:** Context rot is the gradual degradation of the model's working state as the session runs long — surfacing as re-asked questions, constraint drift, incorrect terminology, or vague responses that were precise earlier.

**Why it matters:** Compaction (manual `/compact`, autocompact, or the server-side Compaction API) produces a structured summary, but that summary is generated by the model and inherits its limitations. Implicit context, naming conventions established by example, and nuanced constraints expressed conversationally are the most likely things to be lossy. Recognising the symptoms lets you intervene before they compound. Note: on a 1M-context Sonnet 4.6 / Opus 4.6 workload you may run for a long time before any compaction fires at all — but rot can still set in from sheer volume of intermediate reasoning. The symptoms are the same.

**Symptoms of context rot:**

| Signal | What it means |
|--------|---------------|
| Model re-asks a question you already answered | That answer didn't survive compaction |
| Model violates a constraint you set earlier | The constraint drifted out of active context |
| Model uses a different name for something | Terminology anchor was lost in summarisation |
| Responses become more hedged and general | Model is reasoning from a summary rather than specifics |
| Model repeats work it already did | It's lost track of completed state |

---

**Example 1 — Coding (constraint drift)**

*Scenario:* A developer is building a form validation library. They established early in the session that all validation errors should use error codes (`REQUIRED_FIELD`, `INVALID_EMAIL`) not human-readable strings. After a long session and compaction, the model starts returning `{ error: "This field is required" }` instead of `{ code: "REQUIRED_FIELD" }`.

*WITHOUT:* The developer doesn't notice until they're writing tests. They accept the change, and the inconsistency is now in the codebase.

*WITH:* The developer notices the output changed format. They recognise this as context rot and respond:

```
Stop — I notice you're using human-readable error strings like "This field is required".
We established at the start of this session that all validation errors use error codes
(REQUIRED_FIELD, INVALID_EMAIL, etc.), not strings. Please rewrite the last output
to use the code format.

Reminder: all validation errors throughout this session must use the code format.
```

They restate the constraint and continue.

---

**Example 2 — Data/Analytics (terminology drift)**

*Scenario:* An analyst is building a revenue model. Early in the session they established that "MRR" means monthly recurring revenue from subscription plans only — not one-time purchases. After compaction, the model starts including one-time purchases in MRR calculations.

*WITHOUT:* The analyst doesn't catch it. The MRR figure in the final model is wrong.

*WITH:* The analyst catches that a calculated figure is higher than expected, investigates, and finds the model has changed the MRR definition. They respond:

```
The MRR figure here seems too high. Are you including one-time purchases?

Reminder: in this project, MRR = recurring subscription revenue only.
One-time purchases, setup fees, and professional services revenue must
be excluded from MRR and tracked separately as "non-recurring revenue".

Please recalculate excluding those categories.
```

---

**Example 3 — Content/Research (vagueness signal)**

*Scenario:* A researcher has been building a detailed competitive intelligence report. Early in the session the model was producing precise, sourced claims. After compaction, responses become more hedged and general: "it appears that", "generally speaking", "their strategy seems to focus on".

*WITHOUT:* The researcher accepts the vaguer output, not recognising it as a symptom of context rot.

*WITH:* The researcher recognises the hedging as a signal. They respond:

```
Your last three responses have been significantly more vague than earlier in this
session — using phrases like "it appears" and "seems to".

We have access to the source material via WebFetch. Please re-read the specific
pages before making claims. Don't hedge on claims that can be verified from the
sources. If something can't be verified, say so explicitly — don't soften it
with language that implies speculation about a verifiable fact.
```

### Skill 11: Writing for Compaction

**What it is:** Structuring your messages to survive the compaction process — knowing what the compaction prompt preserves (user messages verbatim, explicit constraints, file names, code snippets) and writing to maximise what survives. Same discipline applies whether compaction is local (`/compact`, autocompact) or the server-side Compaction API.

**Why it matters:** The compaction prompt in `services/compact/prompt.ts` (as of v2.1.128, 2026-05-05) explicitly instructs the summarising model to list all user messages that aren't tool results. User messages are the most durable element in the context. Tool results, intermediate model reasoning, and implicit context established through example are the most volatile. Writing for compaction means putting your important context in user messages, not relying on the model's tool output or intermediate thinking to carry it forward. This survives every variant — manual `/compact`, autocompact, the Compaction API, and re-summarisation across forked-skill contexts.

**Writing for compaction means:**
- State constraints as standalone messages, not as replies embedded in conversation flow
- Name things explicitly — don't establish terminology only through example
- Write "confirmed finding:" or "decision for this session:" to signal a message that should survive
- Reference specific file paths and function names in your messages, not just in tool calls

---

**Example 1 — Coding (constraint durability)**

*Scenario:* A developer is implementing a caching layer. The business rule is that cache TTLs must be configurable per-environment and must never be hardcoded.

*WITHOUT:* The developer establishes this rule by commenting on the model's first output: "Don't hardcode that — it should be configurable". This is a conversational reply, and during compaction the nuance may not be captured. Later the rule is violated.

*WITH:* The developer writes the constraint as a standalone user message, structured for durability:

```
Architecture constraint for this entire session:

Cache TTLs must NEVER be hardcoded in source files.
All TTL values must be read from environment variables or config files at runtime.

This applies to every cache implementation we build today. If you're about to
hardcode a TTL value, stop and ask me for the config key name instead.
```

A standalone message with clear structure. The compaction prompt's "all user messages" section will capture this.

---

**Example 2 — Data/Analytics (finding preservation)**

*Scenario:* A data engineer has spent several turns exploring a data quality issue and has identified that the root cause is duplicate `user_id` rows in the `stg_users` model caused by a specific ETL job. They need this finding to survive compaction.

*WITHOUT:* The finding exists in the model's tool output and in the engineer's conversational responses. It might survive compaction, or it might be compressed to "investigated data quality issue".

*WITH:* The engineer writes a consolidating message:

```
Checkpoint — confirmed finding for this session:

Root cause identified: duplicate user_id rows in stg_users are caused by the
Salesforce ETL job (dbt/models/staging/stg_users.sql) not deduplicating on
import. The ETL runs twice daily and the second run inserts duplicates when
Salesforce records are updated.

Next step: add a QUALIFY ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY
updated_at DESC) = 1 filter to stg_users.sql.

This is the confirmed cause — no need to re-investigate.
```

This message is explicit, references specific file names, and states the next step. It will survive compaction in a useful form.

---

**Example 3 — Content/Research (decision preservation)**

*Scenario:* A writer and model have gone through several rounds deciding to frame a report around three themes rather than five. The decision was reached through dialogue and exists in the conversation flow but was never stated as a clean decision.

*WITHOUT:* The framing decision exists implicitly in the dialogue. After compaction the model might return to suggesting the five-theme structure it initially proposed.

*WITH:* The writer writes a consolidating message:

```
Decision for this document — confirmed:

We are using three themes, not five:
1. Speed to value
2. Integration simplicity
3. Compliance without complexity

The original five-theme structure has been rejected. Do not propose adding
"cost efficiency" or "scalability" back as themes. If you think a point
belongs in the document, fit it into one of the three existing themes.
```

### Skill 12: Checkpointing — and choosing the right primitive

**What it is:** Forcing a context consolidation on your own terms — before the system does it automatically — by explicitly asking the model to summarise the current state, then verifying the summary is accurate before continuing. In 2026 this is the *first* of several context-management primitives; the skill is knowing when to reach for which.

**Why it matters:** Auto-compaction in Claude Code fires when the context window hits a threshold (approximately `contextWindowSize - AUTOCOMPACT_BUFFER_TOKENS`, as defined in `autoCompact.ts` — symbol referenced as of v2.1.128, 2026-05-05; not re-verified). You have no control over when it fires or what it preserves. A 1M-context-autocompact false-positive fix shipped in v2.1.128, but the underlying mechanism is unchanged: when it fires, you wear the lossy summary. Manual checkpointing gives you both control over timing and the ability to verify the summary — you can correct drift before it's baked into the continued session context. A corrected checkpoint costs one turn. Uncorrected drift costs many.

**The decision tree — manual checkpoint vs `/compact` vs Compaction API vs Memory vs forked-skill context:**

- **Manual checkpoint message.** You want to consolidate state *and verify it* before continuing. Cheapest move; one turn. Use mid-task, before a hard sub-step, or whenever you suspect rot has begun. This is the default for any session you're driving by hand.
- **`/compact` (Claude Code).** The window is filling, you're not at a clean stopping point, and you'd rather force the lossy summary on your own timing than the harness's. Use when a manual checkpoint isn't enough — you need to actually reclaim window space, not just verify state.
- **Compaction API (Anthropic API, Opus 4.6, beta as of 2026-05-05).** You're building an API workload — agent, batch, or long-running service — and you want the platform to manage rolling summaries instead of you hand-rolling `/compact`-equivalent logic. The shape is the same as local compaction; the lifecycle is server-side. Reach for this when there's no Claude Code harness in the loop and the conversation is genuinely long-horizon.
- **Memory (Managed Agents, public beta as of 2026-05-05).** State that should outlive *this* session. File-backed, scoped, audited. Confirmed findings, project conventions, preferences that apply across runs. CLAUDE.md still covers static project instructions; Memory covers state the agent itself learns. (Covered in depth in Layer 3.)
- **Forked-skill context (`context: fork` in `SKILL.md`, v2.1.126+).** A skill needs clean working memory and/or deferred tools (WebSearch, WebFetch are deferred and only available to forked-context skills on first turn). Use when a sub-task is conceptually distinct enough that it shouldn't pollute the parent context — e.g. a research skill doing a multi-fetch sweep that you don't want compacted alongside your main thread.

These compose. A long agentic build might use Memory for persistent project conventions, the Compaction API for in-run context management, and a forked-skill context for an embedded research subtask — all in the same session.

---

**Example 1 — Coding (mid-refactor checkpoint)**

*Scenario:* A developer is halfway through a multi-file refactor. They've completed five files and are about to start on the most complex one.

*WITHOUT:* The developer continues. The session eventually compacts. The compaction summary gets the completed files right but misses a constraint established in turn 12: that all event names must be prefixed with the module name. The remaining files are written without the prefix.

*WITH:* Before starting the complex file, the developer asks:

```
Before we continue, give me a checkpoint summary:
1. Which files have we completed in this refactor, and what was changed in each?
2. What constraints are we working under for this refactor?
3. What's left to do?

I'll confirm the summary is correct before we continue.
```

The model produces the summary. The developer checks it. If the event naming constraint is missing, they add it:

```
Good summary, but you've missed one constraint: all event names must be prefixed
with the module name (e.g., cart.item_added, not item_added). Add that to the
constraints and continue.
```

If the constraint is one that should outlive *this* session (a project convention, not a one-off refactor rule), the developer also writes it to Memory or to `CLAUDE.md` — checkpoint solves the in-session problem; Memory / CLAUDE.md solve the across-session problem.

---

**Example 2 — Data/Analytics (pipeline build checkpoint)**

*Scenario:* A data engineer has been building a complex dbt pipeline and is about to start the final mart layer models.

*WITHOUT:* The engineer continues. The model builds the mart layer with slightly wrong assumptions about the staging layer's grain (order line level vs order level) because the grain was established through a tool output that compacted lossy.

*WITH:*
```
Checkpoint before we build the mart layer. Summarise:
1. The grain of each staging model we've built (what one row represents)
2. The key columns available for joining between staging models
3. The business rules we've established for how to handle returns and cancellations

If any of these are unclear from our conversation, say so rather than guessing.
```

The engineer reviews and corrects before proceeding. If the pipeline is large enough that the window is also under pressure, they follow the checkpoint with `/compact` to reclaim space — checkpoint first (verify state), compact second (reclaim window). Doing it in that order means the corrected understanding is what gets summarised, not the pre-correction state.

---

**Example 3 — Content/Research (document checkpoint with forked research)**

*Scenario:* A researcher and model have been building a long report. They're at section 4 of 7. Section 5 needs a fresh round of WebFetch-driven competitor research that they don't want polluting the main thread.

*WITHOUT:* They continue into section 5 in the same context. The web research adds hundreds of intermediate fetches; autocompact fires during section 5; the summary loses the specific framing decision made in section 2 and the model continues with inconsistent framing.

*WITH:* Before starting section 5, the researcher checkpoints the main thread:

```
Checkpoint before section 5:
1. Summarise the core argument of each completed section (1-4) in one sentence each
2. What is the agreed-upon framing for this report (we discussed this early on)?
3. Are there any style rules or constraints we established that I should re-confirm?
```

The researcher verifies, corrects if needed. Then they push the actual web research out into a skill with `context: fork` — the deferred WebFetch / WebSearch tools become available on first turn, the research happens in its own sub-context, and only the consolidated findings come back into the main thread. The main thread keeps its framing intact; the research thread doesn't have to.
