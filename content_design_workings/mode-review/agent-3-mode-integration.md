# Agent 3 — Human Mode / Machine Mode Integration Review
*Lens: where does the frame already live, and how do we make it explicit?*
*Date: 2026-04-15*

---

## The Core Finding

The Human Mode / Machine Mode distinction is already latent in the course. It just has no name. Machine Mode is what the course has been teaching throughout — every template, every stopping condition, every safe verb, every constraint, every Opening Seed. Human Mode barely exists in the course content, which is both the problem and the opportunity: naming that absence is what makes the frame land.

The insight the learner is missing is not that there are two different ways to use AI. It is that the reason their AI interactions feel either chaotic or over-controlled is because they do not know when to use which one. The frame resolves a felt confusion the learner already has. Introducing it should feel like someone finally naming what has been happening, not like a new concept being dropped in.

---

## 1. Human Mode Inventory: Where It Already Exists in the Course

Human Mode moments in the course are thin, but they are present. They tend to appear in three places: the "If this didn't work" / "If you're not sure" sections, the guessing-is-fine moments, and the setup for any exercise that uses open real-life material.

**B-02 — "If you are not sure what to put in a bracket"**
> "Guess. Put in your best attempt and send it. The AI will respond based on whatever you wrote, and you can always try again with different words. There is no penalty for getting it wrong the first time."

This is the closest the Beginner track gets to Human Mode — an explicit permission to not know yet, to be exploratory, to let the AI help you discover what you want. It is framed as a practical workaround, not as a mode of use. The reframe: this is what Human Mode looks like. You are thinking out loud with the AI.

**B-04 — "If this did not work" (implied richer context needed)**
> "If both responses looked similar, check whether you personalised the Version B brackets fully."

This section implicitly recognises that the learner may not yet know what they need. The exercise of comparing Version A and B is actually a Human Mode / Machine Mode comparison in disguise: one message that lets the AI guess (closer to Human Mode), one that instructs precisely (Machine Mode). But the module does not name it that way.

**B-07 — The output review checklist**
The checklist asks the learner to read carefully and *decide* what happened: "Did it answer what I actually asked? Not a related question, not a broader version of the question — the specific thing you asked." This exercise in discernment is a Human Mode activity — using your judgement to interpret and respond — but it is not named as such.

**B-08 — Recovery, Step 1: "Help me write something"**
The deliberate vague prompt exercise — "Help me write something" — is the course's most explicit invocation of Human Mode, but as a cautionary example of what not to do rather than a legitimate mode. The exercise shows Human Mode behaviour (vague, exploratory) in order to teach that it produces vague output. This is a missed opportunity to say: there is a time when that kind of message is exactly right.

**B-09 — "If you want to try Human Mode instructions as well"**
This is the single explicit mention of the phrase "Human Mode" in the entire course. It is tucked into the "If this did not work" section at the bottom of the module:
> "If you want to try Human Mode instructions as well — warmer, more conversational, less structured — that is a good next step. Set up a second project or Gem with different preferences. Switch between them depending on what kind of task you are doing."

This is a major moment that gets almost no weight. It introduces the term, acknowledges the distinction, and tells the learner to set up a second project — but it does it in one short paragraph as an aside, not as a concept worth learning. There is no module that teaches what Human Mode actually is, when to use it, or what you do in it.

**E-07 — "Which three principles do you actually use?"**
The reference card exercise invites the learner to reflect on their own practice in plain language. "Not the ones that felt interesting in the module. The ones you actually reach for." This is a Human Mode thinking exercise inside a Machine Mode course — reflective, exploratory, self-directed. It is never named as such.

**A-08 — "Never delegate understanding"**
> "If you need to understand something — to learn it, to develop a view on it, to make a decision based on it — do it in your main session."

This is the clearest articulation of what Human Mode is for, buried deep in the Advanced track in the context of delegation. It says, in effect: some work has to stay with you. The AI is not a substitute for thinking. That is the thesis of Human Mode — but it arrives too late and in the wrong frame.

## 2. Machine Mode Inventory: Where It Already Exists in the Course

Machine Mode is the default mode of instruction throughout. Every template, every COPY — PERSONALISE — USE section, every stopping condition, every safe verb is a Machine Mode skill. The list below marks the defining moments.

**B-02 — The four-verb shortlist (safe verbs)**
Summarise / Explain / Draft / List. These are Machine Mode verbs — each has "a clear ceiling." The framing is practical, not conceptual, but the underlying idea is pure Machine Mode: you are choosing a verb that limits the AI's decision space.

**B-04 — Version B: detailed structured prompt**
The entire point of the module is Machine Mode: give the AI enough specific input to produce a specific output. "Not a magic formula — you gave the AI more to work with." The Without / With structure that dominates the Advanced track begins here.

**B-05 — One Thing at a Time**
Single-purpose messages are Machine Mode discipline applied to message structure: one verb, one target. "Keep each message to exactly one verb and one target."

**B-06 — Tell It When to Stop**
This is the hallmark Machine Mode module. Stopping conditions are the defining feature of Machine Mode: "I know exactly when this is done." The definition of done is the Machine Mode contract. The module makes this explicit: "'Keep it short' is not a stopping condition. 'Under 80 words' is."

**B-09 — The Machine Mode persistent instruction template**
The module explicitly names this: "what you might call Machine Mode defaults." It then immediately drops the terminology for the rest of the module. But the template it produces — no preamble, one question at a time, stop when done — is a Machine Mode operating document. This is the clearest, highest-value opportunity for the frame, and it is currently squandered.

**B-10 — The Opening Seed**
Four-part structure: Role, Task, Context, Done. The "Done when" slot is the Machine Mode marker — it is the definition of done, the stopping condition, the instruction that closes the task. The entire Opening Seed is a Machine Mode instrument.

**E-01 — The First Word Matters**
Safe vs risky verbs. The entire concept of a safe verb is a Machine Mode concept: "Some verbs have a clear ceiling... Others have no ceiling at all." When you use a risky verb, you are not in Machine Mode. When you use a safe verb, you are.

**E-02 — Tell It What to Leave Alone**
Constraints are Machine Mode boundary-drawing. "Same verb. Same task. The second version tells it where its authority ends." Defining authority is the Machine Mode act.

**E-03 — Context Is Fuel**
The load-bearing test is Machine Mode thinking: "would the AI have to guess if this were missing?" The question reframes context not as helpfulness but as precision engineering.

**E-05 — Get the Plan First**
"Don't do any of them yet." The two-message pattern explicitly holds execution until the human has reviewed and approved the plan. This is Machine Mode oversight — the human is in control of every step. But it also has a Human Mode component: the first message (get the plan) is a moment of thinking-together before switching to execution.

**A-03 — CLAUDE.md**
A Machine Mode document. Standing rules, explicit constraints, named conventions. Everything that survives compaction is Machine Mode writing: "explicit beats implicit, standalone beats buried, named beats described."

**A-04 — Context Awareness and Compaction**
The entire module is about how to write Machine Mode messages that survive the session. The compaction survival checklist is a Machine Mode discipline checklist.

**A-06 — Plan Before Execute**
The plan / execute separation is the Machine Mode workflow made explicit in Claude Code. "Do not start working until I confirm the plan." Machine Mode is not just about the prompt — it is about the workflow.

**A-08 — The 5-Part Brief**
A Machine Mode document in its most structured form. Five named parts, each with a specific function. "Write a brief that could be handed to someone with no other context and still produce exactly what you need." This is Machine Mode writing at its most developed.

## 3. The Transition: Where It Exists, Where It Is Missing

**Where it exists (partially):**

The closest the course comes to teaching the transition is E-05: Get the Plan First. The two-message pattern is structurally a transition moment: Message 1 is exploratory (what would you do? — the AI proposes, the human evaluates), and Message 2 is executory (go ahead — Machine Mode). But this is not named as a transition between modes. It is framed as a technique for avoiding mid-task corrections, not as a way of moving from not-knowing to knowing-enough-to-instruct.

B-08 Recovery has a latent transition in it: the restate exercise. You write "Help me write something" (Human Mode: vague, open), fail, then write a precise structured restate (Machine Mode: role, task, stopping condition, constraint). The learner moves through both modes in the exercise. But again, the transition is not named. It is presented as a correction technique, not a mode-switching skill.

B-09's aside about Human Mode instructions is the closest to explicitly naming the transition: "Switch between them depending on what kind of task you are doing." But this is one sentence at the bottom of a module. There is no guidance on how to know which kind of task you are doing, or when to switch, or what the switch looks like in practice.

**Where it is missing:**

There is no module — and no substantial section within any module — that:
1. Names the transition as a skill
2. Teaches how to recognise which mode a task currently calls for
3. Shows what "I now know enough to give precise instructions" looks like
4. Demonstrates moving from an open-ended exploration to a Machine Mode seed

The course implicitly assumes that learners always know what they want before they start. The entire B-track is built for a learner who has a task and needs to instruct clearly. But many learners arrive with "I think I want to do something with this but I'm not sure what" — and the course has no home for that state. Human Mode is where that thinking happens. The transition is the moment when the thinking produces enough clarity to shift into Machine Mode.

## 4. Where to Introduce the Frame

### Primary introduction point: B-09, mid-module

B-09 currently uses the phrase "Machine Mode defaults" once, then drops it. It should be the place where both modes are named and explained — not as an aside but as the module's central concept. The module is already about setting up persistent instructions; the natural upgrade is to explain that you are choosing which mode to default to, and that there are two.

The introduction should land after the "What is happening" section and before "The exercise." It is introduced as a revelation of what the learner has been doing since B-01.

**Draft language for B-09 introduction of the frame:**

> Since B-01, you have been learning a specific way of working with AI. You give it a role, a task, a constraint, a stopping condition. It executes. You review. This approach has a name: Machine Mode. It is precise, structured, and predictable. It is how you get the AI to do exactly what you asked and stop.
>
> There is another way of working with AI. You have already used it, probably without naming it. It is the way you work when you are not yet sure what you want — when you want to think something through, explore options, get reactions to a half-formed idea, or figure out what you actually need before you start giving precise instructions. That is Human Mode. Conversational. Exploratory. Less structured. The AI is a thinking partner, not a machine taking orders.
>
> Neither mode is better. They are different tools for different moments.
>
> The machine you have been setting up in this module — the persistent instructions that cut out preamble, skip next steps, and get straight to the point — is a Machine Mode document. It tells the AI how you want it to behave when you are ready to work, when you know what you want, when you need it to execute cleanly and stop.
>
> But not every task starts there. Some tasks start with "I'm not sure what I want here" or "Help me think through this." For those tasks, you want a different setup — warmer, more conversational, willing to ask questions and explore. If you want to try that mode, set up a second project with different instructions.
>
> The skill this course is building toward is knowing which mode a moment calls for — and how to switch.

### Secondary introduction point: B-04 intro

B-04 ("Your Input Shapes the Output") is the first module that systematically shows the difference between a vague message and a precise one. The comparison between Version A and Version B is implicitly a Human Mode / Machine Mode comparison. Adding a one-paragraph framing here — before the exercise — would plant the seed several modules before the formal introduction.

**Draft language for B-04 (light touch):**

> Notice what you are about to do: you are going to send the same request twice — once loosely, once precisely. The first version is what you might send when you are exploring, when you are not sure exactly what you need back. The second is what you send when you know. Both are valid. This module shows you what the difference produces.

### Tertiary introduction point: E-05 transition acknowledgement

E-05 (Get the Plan First) should explicitly name itself as a transition technique. The two-message pattern is structurally the move from "let's see what the AI thinks" (Human Mode input) to "now go ahead" (Machine Mode execution). One sentence of naming this would make the module significantly more resonant.

**Draft language for E-05 concept section (addition):**

> This two-message pattern is a deliberate transition. The first message is exploratory — you are asking the AI to show you its thinking so you can evaluate it. The second is executory — you are giving a precise instruction based on what you just learned. This is the switch from thinking-with-AI to directing-AI. Once you see it this way, you will start doing it naturally for any task where you are not sure exactly how it should go.

## 5. Reinforcement Map: Module by Module

**B-01 — Your First Message**
No change needed. This module is deliberately mode-neutral — it is just about sending one message and not being scared. The mode frame does not belong here.

**B-02 — Copy, Personalise, Use**
Light touch. The "If you are not sure what to put in a bracket — guess" section should gain one sentence: "That kind of exploratory message — where you are not sure yet, and you are letting the AI help you figure out what you need — is actually a legitimate way to start. We will come back to that. For now, guess and send."

**B-04 — Your Input Shapes the Output**
Add the light-touch framing above. The Without/With structure is already a Mode A / Mode B comparison. Naming it this way once (without the formal terms yet) primes the learner for B-09.

**B-05 — One Thing at a Time**
No framing needed. This is structural Machine Mode discipline, best kept practical.

**B-06 — Tell It When to Stop**
Add one sentence at the end of "What good looks like": "A response that stopped exactly where you asked it to stop is Machine Mode working as it should. You gave it a definition of done and it used it."

**B-07 — What Just Happened?**
The output review checklist is a Human Mode activity — it requires judgement, reading, and decision. Add one sentence at the end of the exercise intro: "Reading the response carefully before reacting is the human part of this. The AI executed; now you evaluate. That is how the two sides of this relationship work."

**B-08 — Recovery**
This module should acknowledge the mode transition explicitly. The restate exercise moves the learner from a vague Human Mode message ("Help me write something") through the failure of that message, to a precise Machine Mode restate. Name the arc. Add a brief note after Step 3: "What you just did is the core transition: you started with something vague and exploratory, got feedback about what did not work, and used that to write something precise. That movement — from not-knowing to knowing — is how Human Mode and Machine Mode work together."

**B-09 — Your Two Projects (Persistent Instructions)**
This is where the frame lives. Full introduction as drafted above. This is the highest-leverage module for the frame.

**B-10 — The Opening Seed**
The Opening Seed is a Machine Mode instrument. Name it briefly in the intro: "The Opening Seed is your Machine Mode starting point. When you know what you want and you are ready to instruct clearly, this is how you do it. If you are not sure yet, you are still in thinking mode — and that is fine. Use a freer conversation to get there, then come back to this structure once you know."

**E-01 — The First Word Matters**
The safe/risky verb distinction maps directly onto the modes. Safe verbs are Machine Mode verbs — they have a defined ceiling. Risky verbs ("improve", "enhance") are closer to Human Mode instructions — you are delegating judgement rather than specifying an output. Add one sentence in the concept section: "When you use a safe verb, you are working in Machine Mode — you are giving the AI a bounded task. When you use a risky verb, you are effectively asking the AI to decide what 'better' means. Sometimes that is what you want; most of the time, it is not."

**E-02 — Tell It What to Leave Alone**
Constraints are the Machine Mode boundary-drawing instrument. Add a one-liner to the concept: "Adding a constraint is a Machine Mode act. You are defining the edge of the AI's authority."

**E-03 — Context Is Fuel**
The load-bearing test is Machine Mode thinking applied to information selection. No framing needed — the module is already precise and well-structured. A brief acknowledgment at the start that this module builds the Machine Mode setup before execution is sufficient: "Everything you add before the AI starts is your Machine Mode setup."

**E-04 — Restate, Don't Patch**
The correction spiral is a Machine Mode failure mode — the original task was not precisely enough specified, so corrections multiply. The restate is the Machine Mode reset. "The restate is not a workaround. It is what a clean Machine Mode opening looks like when you finally know what you actually need."

**E-05 — Get the Plan First**
Add the transition framing as drafted above. This is the structural transition moment in the E-track.

**E-06 — Migration and Re-seed**
The re-seed is Machine Mode writing applied to conversation management. No framing needed beyond what is already there. A brief mention that the re-seed is a Machine Mode instrument — carrying only load-bearing context forward — reinforces the frame without adding length.

**E-07 — Your Reference Card**
The reference card is a Machine Mode document. Add one sentence in the intro: "The card you are about to build is a Machine Mode reference. It is for when you know what you need and want to instruct clearly, quickly. You will learn to build the Human Mode equivalent intuitively — it needs no card, because it is just thinking out loud."

**A-01 — Mental Model Refresher**
"Claude Code executes your message, not your intent" is a Machine Mode principle stated at the highest level of the course. It is worth acknowledging the mode frame here: "Everything you learned in the Beginner and Enabled User tracks was preparation for this principle. Precision in your message is Machine Mode thinking. Claude Code rewards Machine Mode more directly than a chat interface does, because the stakes are higher — it is not just words in a window, it is changes to real files."

**A-03 — CLAUDE.md**
CLAUDE.md is a Machine Mode document in its most durable form. The introduction should say so: "The persistent instructions you set up in B-09 for your chat tool have an equivalent here. CLAUDE.md is your Machine Mode contract with Claude Code. What you write in it is the standing instruction layer — the baseline from which every session starts."

**A-04 — Context Awareness and Compaction**
"Explicit beats implicit, standalone beats buried, named beats described" is a Machine Mode writing principle. The compaction survival checklist is a Machine Mode writing checklist. Name it as such in the intro: "Writing for compaction is Machine Mode discipline applied to session management. The principles are the same: be explicit, be specific, name rather than describe."

**A-06 — Output Engineering**
Plan before execute is the Machine Mode workflow formalised. The /plan command is a Machine Mode tool. "This module is about taking deliberate control of what Claude Code produces before it starts — the Machine Mode approach to session management."

**A-08 — Agent Delegation**
"Never delegate understanding" is the highest-stakes Human Mode principle in the course. It deserves a brief note connecting it to the frame: "Delegation is a Machine Mode tool — you give a brief, the agent executes. But the thinking that precedes the brief, the judgement that evaluates the output, the decision about what the task actually needs: those are Human Mode work. They cannot be delegated. That is the line."

## 6. Minimum Viable Addition

The frame can be threaded through the existing course with two structural additions and six small edits. No new module is needed, but one new module would significantly strengthen the frame at the course level. Here is both options.

### Option A: Thread only (no new module)

The minimum viable addition is a rewrite of B-09 to make it the frame's home. The change requires:

1. **B-09 expanded introduction** — add the 4-paragraph "Since B-01..." framing block drafted above (~250 words). This introduces both modes, names them, and explains the B-09 persistent instruction as a Machine Mode document. Also expands the Human Mode aside from one sentence to a proper paragraph.

2. **B-08 transition acknowledgment** — add a 2-sentence note after the restate exercise naming the arc from vague to precise as the mode transition. This gives the learner a felt experience to attach the concept to before the formal naming in B-09.

3. **B-04 light touch** — add a single sentence before the exercise acknowledging that Version A is the exploratory starting point, Version B is the precise instruction. (~30 words)

4. **E-05 transition naming** — add one paragraph after the concept section naming the two-message pattern as an explicit mode transition. (~60 words)

5. **B-09 template rename** — the template is currently called "the Machine Mode persistent instruction template." The label is right but currently unexplained. With the expanded introduction, the label lands properly.

6. **B-10 one-liner** — add a sentence naming the Opening Seed as a Machine Mode instrument and acknowledging that some tasks need a freer starting point first. (~40 words)

These six changes together make the frame land across the course without requiring any new modules or significant restructuring. Total addition: approximately 400-500 words of new content, mostly concentrated in B-09.

The downside: Human Mode remains poorly served. There is still no exercise that teaches a learner to work in Human Mode — to explore, think out loud with the AI, use it to develop clarity rather than execute tasks. The frame is named, but only one of its two sides is taught.

### Option B: Thread + one new module (recommended)

Add a short module — approximately E-01.5 or positioned between B-09 and B-10 — that teaches Human Mode as a mode of use, not just names it. The module has one job: show the learner that exploratory, conversational, non-template AI use is legitimate and productive, that it serves a different purpose than Machine Mode, and that knowing when to use each is the real skill.

The module's exercise: start a conversation with the AI using a genuinely open question about something real in the learner's work or life. No template. No stopping condition. No verb constraint. Just think out loud. Then, at the end of the conversation, write a Machine Mode Opening Seed based on what the exploration clarified.

That exercise — done once, with both steps — teaches the transition experientially. The learner feels the difference. The frame becomes a description of something they just did, not a concept they memorised.

**Working title:** "Think First, Then Instruct" or "Before the Seed" (if positioned before B-10).

**Module objective:** After this, you can identify when you are in thinking mode and when you are in execution mode — and you have used AI productively in both.

**Placement options:**
- Between B-08 and B-09 (before the persistent instructions setup, so the learner understands what mode they are setting up defaults for)
- Between B-09 and B-10 (after the frame is introduced, as the practical exercise)
- As a standalone module at the E-track entry point (E-00 or introductory framing for E-01), positioned as a stage threshold concept

The strongest placement is between B-09 and B-10. B-09 introduces the frame and sets up Machine Mode defaults. The new module teaches Human Mode as a complement. B-10 (the Opening Seed) becomes the capstone of the beginner section as the Machine Mode instrument — the learner understands exactly what they are building and why.
